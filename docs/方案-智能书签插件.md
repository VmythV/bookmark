# 方案（最终定稿）：智能书签插件

## 目标与范围

一键收藏当前网页 → 本地 RAG 召回 + 云端 LLM 精排，智能推荐放入哪个文件夹或建议新建 → 写入浏览器原生书签；支持单向备份/导入导出到 WebDAV/S3；并提供**对已有书签的一键重组**（自动聚类成新的文件夹结构）。

**范围内**：Chrome/Edge(MV3) · WXT+TS · 悬浮按钮**+**工具栏弹窗双入口 · 多语言本地嵌入(`multilingual-e5-small`) RAG · 云端 `/v1/chat/completions` 精排 · 单向备份(手动+定时, 覆盖) · HTML 导入导出 · **书签重组（嵌入+HDBSCAN 聚类）**。

**范围外**：Firefox/Safari · 双向同步 · WebLLM/本地 LLM · Responses API · 自建后端 · 凭据加密（本期明文）。

## 关键决策（含原因）

1. **浏览器书签=主存储**：`chrome.bookmarks` 写入即原生书签，"联动"为天然结果。`storage.local` 只存配置/API Key/向量索引/备份元数据（**明文，仅本地不上传**）。
2. **本地 RAG 用 Domicile 骨架，嵌入模型换 `multilingual-e5-small`**：Domicile 已集成 Transformers.js 嵌入 + HNSW + IndexedDB，集成最快；换多语言模型保证**中英文质量都好**。文件夹名/标题在本地变向量，不外发。
3. **LLM 只做 `/v1/chat/completions` + `response_format: json_schema(strict)`**：端点不确定支持 Responses API，chat 兼容性最广，结构化输出零解析。**不纳入 WebLLM**；无 Key/离线时回退为纯本地关键词规则推荐。
4. **收藏推荐两段式**：本地 HNSW 召回 Top-K 候选 → LLM 精排返回 `{action:"use_existing"|"create_new", folderId?, newFolderPath?, confidence, reason}`。
5. **单向备份=整体覆盖快照**：WebDAV(PUT/GET) + S3(SigV4 PUT/GET) 适配器，远端存标准 HTML 书签文件。触发方式：**手动按钮 + 定时**（`chrome.alarms`）。
6. **书签重组=嵌入 + HDBSCAN 聚类 + LLM 命名**：全部书签嵌入 → HDBSCAN 自动聚类（无需预设分类数）→ 每个簇交 LLM 起文件夹名 → 生成"重组预览"供用户确认后再落地。借鉴 Vectoria 的 `umap-js`/`hdbscan-ts` 技术组合（仅借鉴依赖，不整包引入）。

## 架构

```
WXT 扩展 (MV3, Chrome/Edge)
├─ content script ── 悬浮按钮 (createShadowRootUi)
├─ 工具栏 popup ──── 收藏面板（与悬浮按钮共用 UI）
├─ 重组页 (options/独立页) ── 聚类预览 + 确认应用
├─ background (service worker) ── 编排
│   收藏流: getTree → 本地嵌入(e5) → HNSW 召回 Top-K → 云端 LLM 精排 → 确认 → bookmarks.create
│   重组流: 全量嵌入 → HDBSCAN 聚类 → LLM 命名 → 预览 → 用户确认 → 批量 move/create
│   备份流: 序列化 HTML → WebDAV/S3 PUT（手动 / chrome.alarms 定时）
├─ 设置页: LLM 端点/Key/模型 · 嵌入模型 · Top-K · 备份目标/凭据/定时周期
└─ storage.local + IndexedDB(向量索引)
        │ fetch                         │ fetch
   云端 OpenAI 兼容端点            WebDAV / S3（HTML 覆盖快照）
   (/v1/chat/completions)
```

**索引维护**：首次全量嵌入入 HNSW；之后监听 `bookmarks.onCreated/onChanged/onRemoved/onMoved` 增量更新。

## 实现步骤

1. WXT+TS 脚手架，权限 `["bookmarks","storage","activeTab","alarms"]` + host_permissions。
2. 悬浮按钮 + popup，共用收藏面板。
3. 书签读写层（getTree / 路径解析 / create / move）。
4. 接入 Domicile，嵌入模型配为 `multilingual-e5-small`，IndexedDB 持久化 + 增量更新。
5. 收藏推荐：HNSW 召回 → chat/completions(json_schema) 精排 → 确认写入。
6. **书签重组**：全量嵌入 → HDBSCAN 聚类 → LLM 命名 → 预览 UI → 批量落地。
7. 备份/导入导出：HTML 序列化/解析；WebDAV、S3(SigV4) 适配器；`chrome.alarms` 定时。
8. 设置页。

## 风险与权衡

- **首次下嵌入模型（multilingual-e5-small，~百 MB 级）**，需进度提示与 IndexedDB 缓存；WebGPU 不可用回退 WASM 更慢。
- **书签重组是破坏性批量操作**——必须"预览 + 确认"再执行，并在执行前自动做一次 HTML 备份以便回滚。
- **S3/WebDAV 浏览器直传需服务端 CORS 正确配置**；S3 需正确实现 SigV4。
- **HDBSCAN 会产生"噪声点"（无法归类的书签）**——需设计兜底（保留原位 / 归入"未分类"文件夹）。
- **LLM 精排把 Top-K 候选+页面标题/URL 发往第三方**，UI 需隐私提示。
- **MV3 service worker 会被回收**，长任务（全量嵌入/重组）需做可中断、可续跑、用 Web Worker 承载计算。

## 不确定的地方（不阻塞开始，可实现中再定）

1. **重组的力度**：是"整库推倒重来"还是"只重组某个子文件夹/未分类项"——建议默认提供"选范围再重组"，而非强制全库。
2. **HDBSCAN 噪声点归属**：默认归入"未分类"文件夹，待确认。
3. **定时备份周期**默认值（如每天一次）——设置页可改。
4. **嵌入模型具体来源/量化版本**（`Xenova/multilingual-e5-small` 等）——实现时定。
5. **重组是否保留原有文件夹结构作为参考**还是完全新建——建议生成新结构、保留旧结构直到用户确认删除。
